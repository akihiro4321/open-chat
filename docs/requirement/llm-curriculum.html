LLM機能開発

フレームワークが隠す部分を、自分の手で書く。会話履歴の組み立て、ツール実行ループ、ストリーミングのパース、メモリの永続化──「Mastraが代わりにやってくれること」を全部自前で実装しながら、LLM機能開発の骨格を身につける構成。

主軸：素のAnthropic / OpenAI SDK 器：汎用チャットアプリ（全部入り） 深掘り①：RAG・検索精度 深掘り②：エージェント設計（マルチ/HITL）

このカリキュラムの方針

素のSDKを主軸にすると、Function Calling のループ、SSEストリームのパース、会話履歴やメモリのデータ設計、トークン管理などを自分で書くことになる。これは遠回りに見えて、どのフレームワーク（Mastra / LangChain / 自社実装）に移っても通用する土台になる。

各章には「学べること」（普遍スキル）と「素のSDKだと自分で書く部分」（フレームワークが隠蔽している実体）を併記した。RAG章とエージェント章を厚めにしてある。

PART Ⅰ ── FOUNDATION
第1部　LLM呼び出しの土台

フレームワーク抜きで「LLMを叩く」ところから。ここを自前で書けると以降が全部読める。

1
LLM機能開発の全体像とSDKの基礎
Chat Completions / Messages API の生の形を知る
  • LLMアプリの構成要素（モデル・プロンプト・コンテキスト・ツール・状態）
  • Anthropic Messages API と OpenAI Chat Completions API の構造比較
  • system / user / assistant ロールと messages 配列の組み立て
  • 温度・max_tokens・stop などパラメータの意味と影響
  • APIキー管理・環境変数・レート制限とリトライ
学べること「LLMは状態を持たない関数」という前提。毎リクエストで全状態を送る必要があるという、後続すべての設計の起点。
素のSDKだと自分で書く部分messages配列の手組み、エラーハンドリング、指数バックオフ、429対応
2
ストリーミング応答の実装
SSEを自分でパースしてチャットUIに流す
  • ストリーミングが必要な理由（体感速度・UX）
  • Server-Sent Events (SSE) の仕組みとイベント種別
  • stream: true のレスポンスを逐次パースする
  • 差分チャンク（delta）の結合とフロントへの中継
  • 途中キャンセル（AbortController）と部分応答の保存
学べることVercel AI SDKのuseChatが裏で何をしているか。SSEの生パースを通じてストリーミングの実体が見える。
素のSDKだと自分で書く部分SSEパーサ、delta結合ロジック、ストリーム中のツール呼び出し検出
3
構造化出力（Structured Output）
LLMの出力をアプリで使えるデータにする
  • 自然言語出力とJSON出力の使い分け
  • JSON Schema / response_format による出力制約
  • Zodによるスキーマ定義と型安全なパース
  • 出力が壊れたときのリトライ・バリデーション戦略
  • 分類・抽出・ルーティングへの応用
学べることLLMを「曖昧な文章生成器」から「信頼できるデータ変換器」にする方法。後のガードレールや評価の前提になる。
PART Ⅱ ── CHAT APP CORE
第2部　チャットアプリの基盤を組む

器となる汎用チャットアプリを立ち上げ、会話を「続けられる」状態にする。

4
チャットアプリの土台構築
Next.js + APIルート + チャットUI
  • Next.js (App Router) でのプロジェクト構成
  • チャットAPIルートの設計（リクエスト→LLM→ストリーム返却）
  • フロント：メッセージ一覧・入力欄・ストリーミング表示
  • サーバー／クライアントの責務分割
学べることBFFパターンでのLLM中継。あなたのNest.js BFF経験がそのまま活きる層。
5
会話履歴の永続化とスレッド管理
「記憶しないLLM」に文脈を持たせる①
  • 会話履歴をDBに保存する設計（thread / message テーブル）
  • スレッド一覧・作成・削除・切り替え
  • 履歴をmessages配列に再構築してLLMに渡す
  • スレッドタイトルのLLM自動生成
  • コンテキスト長の限界と履歴の切り詰め戦略
学べること「メモリ」の正体は履歴のDB保存＋再注入だと体で理解する。フレームワークのMemory抽象が何を隠しているか分かる。
素のSDKだと自分で書く部分スキーマ設計、履歴の組み立て、トークン超過時のトリミング
6
マルチモデル対応とフォールバック
モデルを差し替え可能にする
  • プロバイダー差を吸収する抽象レイヤーの自作
  • 用途別モデル選択（高速・高品質・安価）
  • エラー時のフォールバックとリトライ
  • コスト・レイテンシのトレードオフ設計
学べることあなたが実務でやっている「実装はDeepSeek、レビューはGPT-5.4」のようなモデルルーティングを、アプリ層で実装する勘所。
PART Ⅲ ── RAG（深掘り）
第3部　RAGと検索精度

チャットに「社内ドキュメントを参照する」能力を足す。検索精度に最も紙幅を割く部。

7
RAGの基礎と最小実装
検索拡張生成のパイプラインを一通り通す
RAG 深掘り
  • RAGが必要な理由（最新性・社内知識・ハルシネーション抑制）
  • パイプライン全体像：チャンク化→埋め込み→格納→検索→生成
  • ベクトルDBの選択（pgvector / LanceDB / Qdrant 等）
  • 埋め込みモデルの選択と次元数
  • 取得した文脈をプロンプトに注入する
学べることRAGは魔法ではなく「検索結果をプロンプトに貼るだけ」。だからこそ検索の質が全てだと分かる。
8
チャンク化と埋め込みの設計
検索精度はここで7割決まる
RAG 深掘り
  • チャンク戦略（固定長 / 文単位 / Markdown構造 / セマンティック）
  • チャンクサイズとオーバーラップのチューニング
  • メタデータ付与（出典・章・更新日）とフィルタ検索
  • 日本語処理の注意点（トークナイズ・CJK）
  • 埋め込み時のprefix（E5のpassage:/query:等）
学べることあなたのDevVault（LanceDB + BM25 + E5）の知見と直結。チャンク設計の良し悪しが検索ヒット率に直接効くことを実測する。
9
検索精度の向上：ハイブリッド検索とリランク
ベクトル検索だけでは足りない
RAG 深掘り
  • ベクトル検索の弱点（固有名詞・略語・完全一致）
  • BM25（全文検索）との併用＝ハイブリッド検索
  • スコア統合（RRF：Reciprocal Rank Fusion）
  • リランカー（cross-encoder）による並べ替え
  • クエリ拡張・HyDE・マルチクエリ
学べること「なぜ正しい文書が出てこないか」を切り分ける技術。検索精度の改善が体系的にできるようになる。
素のSDKだと自分で書く部分BM25とベクトルのスコア統合、RRF、リランク呼び出しの組み込み
10
RAGの評価と継続改善
精度を「感覚」から「数字」にする
RAG 深掘り
  • 検索評価指標（Recall@k / MRR / nDCG）
  • 生成評価（忠実性 Faithfulness / 関連性 Relevance）
  • 評価データセットの作り方
  • RAGAS等の評価フレームワークの考え方
  • ドキュメント更新時の再取り込み（差分インデックス）
学べること「なんとなく良くなった」を排し、検索改善を定量で回す。あなたのDORA/メトリクス志向と相性が良い。
PART Ⅳ ── AGENT（深掘り）
第4部　エージェント設計（マルチ / HITL）

チャットに「自分でツールを使い、判断し、人間に確認を取る」能力を足す。もう一つの深掘り部。

11
ツール呼び出し（Function Calling）の実装
エージェントの心臓部を自前で書く
エージェント 深掘り
  • ツール定義（名前・説明・パラメータスキーマ）
  • LLMがツールを要求→実行→結果を返すループ
  • 複数ツール・並列ツール呼び出しの扱い
  • ツール実行エラーのLLMへの差し戻し
  • ツールとしてのRAG検索（第3部との接続）
学べること「エージェント＝LLM＋ツール＋ループ」の正体。このループを手で書くと、あらゆるエージェントフレームワークの中身が見える。
素のSDKだと自分で書く部分tool_use/tool_resultのやり取り、実行ループ、最大反復数の制御
12
Human-in-the-Loop（HITL）
危険な操作の前に人間の承認を挟む
エージェント 深掘り
  • HITLが必要な場面（破壊的操作・外部送信・課金）
  • 承認ゲートの設計（実行前に止めて人間に問う）
  • 実行状態の中断と再開（状態スナップショット）
  • 「提案するが適用しない」パターン
  • 承認UIとチャットフローの統合
学べることあなたが既に作っている「propose-don't-apply」（pw-impact-analysis等）の設計思想を、汎用的なワークフロー制御として実装する。
素のSDKだと自分で書く部分実行の中断点、状態の永続化と復元、再開時のmessages再構築
13
マルチエージェント設計
役割を分けて協調させる
エージェント 深掘り
  • 単一エージェントの限界とタスク分解
  • スーパーバイザー（オーケストレーター）パターン
  • サブエージェントへの委譲と結果の集約
  • エージェント間の状態・コンテキスト受け渡し
  • 役割別モデル選択（軽いタスクは安いモデルへ）
学べること「マルチエージェント＝タスク分解」。役割分割の設計判断と、過剰分割の落とし穴を学ぶ。
14
エージェント的ワークフローと制御フロー
自由なエージェントと固いワークフローの間
エージェント 深掘り
  • 「LLMに任せる」vs「処理を固定する」の使い分け
  • 分岐・ループ・並列の制御フロー
  • 信頼性のための明示的なステップ設計
  • 失敗時のリカバリと冪等性
学べることどこをLLMの裁量に任せ、どこをコードで固めるか。プロダクションでの信頼性を左右する設計判断。
PART Ⅴ ── PRODUCTION
第5部　本番運用に耐えるアプリにする

動くものから、出せるものへ。安全・観測・評価・配備。

15
ガードレールと安全対策
入力と出力を守る
  • 入力ガードレール（プロンプトインジェクション対策）
  • 出力ガードレール（PII・有害・機密の検出）
  • ツール実行の権限制御とサンドボックス
  • RAGにおける出典の強制と幻覚抑制
学べることOWASP LLM Top 10 の観点を実装に落とす。あなたのAIガバナンス知見と直結。
16
オブザーバビリティ（LLMOps）
何が起きているか見えるようにする
  • トレースの構造（リクエスト→ツール→生成のスパン）
  • OpenTelemetry / Langfuse 等での計装
  • トークン・コスト・レイテンシの可視化
  • 本番ログから改善点を見つける
学べることあなたがCodex CLIで検討していたLangfuse / Phoenix のOTel計装を、アプリ側で実装する。
17
評価（Evaluation）
変更が改善か劣化かを判定する
  • バッチ評価（データセット）とライブ評価の違い
  • LLM-as-a-Judge とコード/NLPスコアラー
  • 評価データセットとルーブリックの設計
  • n回試行・中央値+IQRでの分散評価
  • 回帰検出：プロンプト変更で壊れていないか
学べることあなたのharness評価設計（n≥5・median+IQR・rubric採点）を、アプリ機能の評価に応用する。
18
認証・課金・デプロイ
サービスとして成立させる
  • 認証・認可（ルート保護、APIキー管理）
  • トークン使用量の追跡とプラン別制限
  • コスト管理とレート制御
  • デプロイ（Vercel / Cloud Run / コンテナ）とCI/CD
  • データ永続化（DB・ベクトルストア）の本番構成
学べることLLMアプリ特有の「従量課金をどう個人/プランに割り当てるか」。あなたのChatGPT Business/Codexシート設計の検討と地続き。
