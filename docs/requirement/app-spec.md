# 汎用AIチャットアプリ 仕様書

> 本書は、LLM機能開発を学ぶためのハンズオン用「汎用チャットアプリ」の仕様書である。
> AIコーディングエージェントへの入力を想定し、「何を作るか／どう振る舞うか」を定義する。
> 詳細設計（テーブルのカラム定義・関数シグネチャ・具体コード）は本書の対象外とし、実装時に決定する。

## 1. プロダクト概要

社内ナレッジに答え、ツールを使い、危険な操作の前には人間に確認を取る汎用AIチャット。

ChatGPT的なチャットUIを器とし、以下を両輪として載せる。

- **RAG**：社内ドキュメントを参照して出典付きで回答する。
- **エージェント**：ツールを使い、複数エージェントで協調し、危険な操作はHITLで止める。

マルチユーザー対応で、ユーザーごとにスレッドと使用量を分離する。プラン別に使用量制限がかかる。

### コアコンセプト：RAGとエージェントの統合方針（制約）

- **§7 で「静的RAG」を実装する。** チャット応答の前段で常に検索を挟み、取得文脈をプロンプトに注入する単純な構成。
- **§11 で静的RAGをツール化し、「agentic RAG」へ発展させる。** RAG検索をエージェントが必要に応じて呼ぶツールの一つとして扱う。
- 最終形は **両モードを切り替え可能**とする。どちらの挙動かはアプリの性格を変えるため、学習対象として明示的に比較できること。

## 3. 機能要件

### 3.1 チャット基盤（§1〜6）

| ID | 機能 | 仕様 |
|----|------|------|
| CHAT-01 | メッセージ送受信 | ユーザー入力をLLMに送り、応答を返す。 |
| CHAT-02 | ストリーミング応答 | 応答を逐次表示する。生成途中のキャンセルに対応する。 |
| CHAT-03 | 会話履歴の永続化 | スレッド単位で履歴を保存し、再開時に文脈を復元する。 |
| CHAT-04 | スレッド管理 | スレッドの作成・一覧・切り替え・削除ができる。 |
| CHAT-05 | スレッドタイトル自動生成 | 会話内容からタイトルをLLMで自動生成する。 |
| CHAT-06 | コンテキスト長管理 | 履歴がモデルの上限を超える場合、古い履歴を切り詰める／要約するなどして送信する。 |
| CHAT-07 | 構造化出力 | 分類・抽出など、出力をJSON等の構造化データとして扱える仕組みを持つ。 |
| CHAT-08 | マルチモデル対応 | 複数プロバイダ／モデルを切り替え可能にする。プロバイダ差は抽象レイヤーで吸収する。 |
| CHAT-09 | フォールバック | モデル呼び出し失敗時に代替モデルへ切り替える。 |

**制約**
- ストリーミングはSSE等のストリームを自前で扱う（高レベルの`useChat`等に丸投げしない方針を学習目的とするが、フロント表示の補助に留めるのは可）。
- 「LLMは状態を持たない」前提を守り、必要な文脈は毎リクエストで構築して送る。

### 3.2 RAG（§7〜10）

| ID | 機能 | 仕様 |
|----|------|------|
| RAG-01 | ドキュメント取り込み | ドキュメントをチャンク化し、埋め込みを生成してベクトルストアに格納する。 |
| RAG-02 | チャンク設計 | チャンク戦略（構造ベース等）・サイズ・オーバーラップを調整可能にする。メタデータ（出典・章・更新日）を付与する。 |
| RAG-03 | ベクトル検索 | クエリの埋め込みで類似チャンクを取得する。 |
| RAG-04 | ハイブリッド検索 | ベクトル検索とキーワード検索（BM25等）を併用し、スコアを統合（RRF等）する。 |
| RAG-05 | リランク | 取得結果を再ランキングして上位を絞り込む。 |
| RAG-06 | 出典付き回答 | 回答の根拠となった文書・箇所を提示する。 |
| RAG-07 | 静的RAGモード | チャット応答の前段で常に検索を実行し、文脈を注入する（§7）。 |
| RAG-08 | 再取り込み | ドキュメント更新時に差分でインデックスを更新できる。 |
| RAG-09 | RAG評価 | 検索評価（Recall@k / MRR 等）と生成評価（忠実性・関連性）を計測できる。評価データセットを用意する。 |

**制約**
- 日本語ドキュメントを扱う前提とし、トークナイズ・CJKの扱いに配慮する。
- 検索精度の改善は「感覚」ではなく**数値（評価指標）**で判断できること（RAG-09 が前提）。

### 3.3 エージェント（§11〜14）

| ID | 機能 | 仕様 |
|----|------|------|
| AGT-01 | ツール呼び出し | LLMがツールを要求→アプリが実行→結果を返す、というループを実装する。 |
| AGT-02 | RAGのツール化 | §7の静的RAG検索を、エージェントが必要時に呼ぶツールとして提供する（agentic RAG, §11）。 |
| AGT-03 | 複数ツール | Web検索・計算など複数ツールを扱える。並列ツール呼び出しに対応する。 |
| AGT-04 | ツールエラー処理 | ツール実行失敗をLLMに差し戻して継続できる。 |
| AGT-05 | 反復制御 | ツール呼び出しループに最大反復数を設け、暴走を防ぐ。 |
| AGT-06 | HITL承認ゲート | 破壊的・外部送信・課金を伴う操作は実行前に停止し、ユーザー承認を待つ。 |
| AGT-07 | 中断と再開 | 承認待ちで実行状態を保存し、承認後に再開する。 |
| AGT-08 | 提案モード | 「提案するが適用しない（propose-don't-apply）」挙動を選べる。 |
| AGT-09 | マルチエージェント | スーパーバイザーがサブエージェントにタスクを委譲し、結果を集約する。 |
| AGT-10 | 役割別モデル選択 | エージェントの役割に応じてモデルを使い分ける（軽い処理は安価なモデル等）。 |
| AGT-11 | 制御フロー | LLMに任せる部分とコードで固定する部分を切り分け、分岐・ループ・並列を制御する。 |

**制約**
- ツール呼び出しループはフレームワーク任せにせず、`tool_use` / `tool_result` のやり取りを自前で書く。
- HITLは「実行前に確実に止まる」ことを保証する。承認なしに副作用のある操作を行わない。

### 3.4 安全・運用（§15〜18）

| ID | 機能 | 仕様 |
|----|------|------|
| OPS-01 | 入力ガードレール | プロンプトインジェクション等の不正入力を検査する。 |
| OPS-02 | 出力ガードレール | PII・機密・有害な出力を検査する。 |
| OPS-03 | ツール権限制御 | ツール実行の権限を制御し、許可された操作のみ実行する。 |
| OPS-04 | オブザーバビリティ | リクエスト→ツール→生成をトレースし、トークン・コスト・レイテンシを可視化する。 |
| OPS-05 | 評価 | バッチ評価（データセット）とライブ評価を行う。プロンプト変更による回帰を検出する。 |
| OPS-06 | 認証・認可 | サインアップ／サインイン、保護ルートを実装する。 |
| OPS-07 | 使用量・課金管理 | トークン使用量を追跡し、プラン別に上限を課す。 |
| OPS-08 | デプロイ | クラウド環境へデプロイし、CI/CDで自動化する。データ永続化を本番構成にする。 |

**制約**
- §15〜18 は全項目の完全実装を必須とせず、**最低1つの運用テーマを本番品質まで仕上げる**ことを優先してよい（学習上の現実的な進め方）。ただしどれを本番品質にするかは実装者が明示すること。

## 5. 振る舞いの定義（主要シナリオ）

### S1. 通常のRAG質問（静的RAGモード）
1. ユーザーが質問を送信する。
2. 質問で社内ドキュメントを検索する（ハイブリッド検索＋リランク）。
3. 取得文脈をプロンプトに注入してLLMに送る。
4. 出典付きで応答をストリーミング表示する。

### S2. エージェントによるツール利用（agentic RAGモード）
1. ユーザーが依頼を送信する。
2. LLMが必要と判断したツール（RAG検索・Web検索・計算等）を呼ぶ。
3. アプリがツールを実行し、結果をLLMに返す。
4. 必要なら2〜3を反復（最大反復数まで）。
5. 最終応答を返す。

### S3. HITLを伴う操作
1. エージェントが副作用のある操作（外部送信・破壊的変更等）を要求する。
2. アプリは実行前に停止し、操作内容を承認UIに提示する。
3. ユーザーが承認 → 実行して継続。却下 → 中止。
4. 承認待ち中も状態を保持し、後から再開できる。

### S4. ガードレール作動
1. 入力／出力がガードレールに抵触する。
2. 該当のリクエスト／応答をブロックし、ユーザーに理由を提示する。

## 6. 非機能要件

| 観点 | 要件 |
|------|------|
| 観測可能性 | 全リクエストをトレース可能にする。トークン・コスト・レイテンシを記録する。 |
| 安全性 | 副作用のある操作はHITLまたは権限制御で守る。インジェクション対策を行う。 |
| 評価可能性 | 機能変更（プロンプト・RAG設定・モデル）が改善か劣化かを数値で判定できる。 |
| 拡張性 | モデル・ツール・データソースを追加できる構造にする。 |
| マルチユーザー | ユーザー単位でスレッド・使用量・認可を分離する。 |

## 7. アーキテクチャ（論理構成・概念図）

```
[ チャットUI ]         ストリーミング表示 / スレッド / 承認UI / 出典表示
      │
[ APIルート・BFF ]     認証 / 入力ガードレール / 使用量チェック
      │
[ エージェント層 ]      ツール呼び出しループ / スーパーバイザー / HITL中断・再開 / 制御フロー
      │
  ┌───┴─────────────────┐
[ RAG検索 ]              [ LLMプロバイダ抽象 ]
ベクトル + キーワード      Anthropic / OpenAI
+ RRF + リランク          モデルルーティング / フォールバック
      │
[ 永続化 ]              会話履歴 / スレッド / 使用量 / ベクトルストア
      │
[ 観測・評価 ]          トレース（トークン/コスト/レイテンシ）/ 評価データセット
      │
[ 出力ガードレール ]     PII / 機密 / 有害の検査
```

※ 上記は論理構成であり、具体的な技術選定・モジュール分割・テーブル設計は設計フェーズで決定する。

## 8. データの種類（概念レベル）

## 10. スコープ外（このアプリで作らないもの）

- 高度な管理者ダッシュボード、組織・チーム管理
- 本格的な決済サービス連携（プラン管理は使用量制御の学習範囲に留める）
- モバイルネイティブアプリ
- 音声・画像入出力（テキストチャットを基本とする。必要なら拡張）
- 高可用性・大規模スケールを前提としたインフラ設計

## 11. 実装上の注意（AIエージェントへの指示）

- **段階的に作る。** §1から順に、前章の成果物に積み上げる。一度に全機能を作ろうとしない。
- **フレームワークに隠蔽させない。** ストリーミング・ツール呼び出しループ・履歴管理・メモリは自前で実装する（学習目的のため）。
- **最新の公式ドキュメントを参照する。** SDK・ライブラリのAPI仕様は変化が速い。実装時に各公式ドキュメントで最新仕様を確認すること。本書はAPIの最新仕様を保証しない。
- **不確かな点は推測で補完せず確認する。**
- **静的RAG → agentic RAG の発展方針（§1のコアコンセプト）を崩さない。**
